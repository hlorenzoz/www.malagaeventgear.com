# Spec: blog-redirects

## Capability

Preserve SEO link equity by emitting explicit, edge-level HTTP 301 redirects from every old WordPress root-level post URL (`/<slug>/`) to the new SvelteKit blog URL (`/blog/<slug>/`), without disturbing existing root-level pages.

---

## Context

The live WordPress site uses a **root-level permalink structure**: posts live at `https://malagaeventgear.com/<slug>/` (verified against the WP REST API `link` field). The new SvelteKit blog serves the same content at `/blog/<slug>/`. Existing root-level pages (`/`, `/packages/`, `/about-us/`, `/contact-us/`, etc.) MUST NOT be redirected. The site runs on Cloudflare via `@sveltejs/adapter-cloudflare`, which supports a `static/_redirects` file evaluated at the edge before the application.

---

## Delta: What MUST Be True After This Change

### 1. Redirect File (`static/_redirects`)

- The file `static/_redirects` MUST contain one redirect rule per migrated blog post.
- Each rule MUST follow the Cloudflare Pages format:

  ```
  /old-path/  /blog/<slug>/  301
  ```

- Trailing slashes MUST be consistent with the site's `trailingSlash: 'always'` configuration: both the source path and the destination path MUST end with `/`.
- The redirect type MUST be `301` (permanent), not `302`, so search engines transfer link equity.
- A single wildcard redirect from `/*` to `/blog/*` MUST NOT be used; rules MUST be listed explicitly per post to prevent matching root-level pages.

### 2. Path Derivation (Pure Function)

- The migration pipeline MUST derive each old path from the WP REST API `link` field by **stripping the origin** (`https://malagaeventgear.com`) and retaining only the path segment (e.g. `https://malagaeventgear.com/my-post/` → `/my-post/`).
- This derivation MUST be implemented as a **pure function** (`deriveOldPath(link: string): string`) with no side-effects, so it is independently unit-testable.
- The function MUST handle trailing slashes consistently: if the WP `link` value does not include a trailing slash, the function MUST append one to match the site convention.
- The function MUST throw (not silently produce an empty string) when the input is not a valid absolute URL belonging to `malagaeventgear.com`.

### 3. Generation by the Migration Script

- The `static/_redirects` file MUST be generated by the existing migration pipeline (`scripts/migrate-wp.ts` or equivalent), which already holds the full post list with real `link` paths from the WP REST API.
- Generation MUST be **idempotent**: running the migration script multiple times MUST produce the same `_redirects` output for the same input post list.
- If `static/_redirects` already exists and contains rules **not** matching any migrated post slug (unrelated rules), those lines MUST be preserved. The generator MUST merge redirect rules, not overwrite the entire file.
- Blog-post redirect rules MUST be clearly delimited by marker comments so the merge logic can identify and replace only the managed block:

  ```
  # BEGIN wp-blog-redirects (managed — do not edit manually)
  /my-post/  /blog/my-post/  301
  # END wp-blog-redirects
  ```

### 4. Taxonomy Archive Redirects (SHOULD)

- If WordPress category archive URLs of the form `/category/<cat-slug>/` are confirmed to exist during migration (verified via WP REST API `_links.wp:term` or direct URL probe), the generator SHOULD emit redirect rules mapping those URLs to `/blog/category/<cat-slug>/`.
- Tag archive redirects (`/tag/<tag-slug>/`) follow the same SHOULD-level requirement.
- Taxonomy redirects MUST also be placed inside the managed marker block.
- If no category/tag archives are confirmed, this requirement MAY be deferred to a follow-up change.

### 5. Edge Evaluation Order

- Because Cloudflare Pages evaluates `_redirects` **before** routing to the SvelteKit application, the 301 redirect for an old post URL MUST be returned at the CDN layer without the SvelteKit app processing the request.
- The generated rules MUST NOT conflict with Cloudflare's 100-rule limit for the free tier. At 75 posts (plus optional taxonomy entries) the total MUST remain below 100 rules; if taxonomy redirects push the count over 100, the generator MUST emit a warning to stdout and taxonomy redirects MUST be omitted from that run.

### 6. Post-Deploy Verification

- After deployment, a verification step MUST confirm that each old post URL returns HTTP 301 with a `Location` header equal to the new `/blog/<slug>/` path.
- The redirect target (`/blog/<slug>/`) MUST itself return HTTP 200.
- This verification MUST be implementable as a script or Playwright check that iterates the generated redirect rules and asserts both conditions.

---

## Acceptance Scenarios

### SC-BR-01: Old post URL returns 301

**Given** the site is deployed with a `static/_redirects` containing the rule `/my-post/  /blog/my-post/  301`  
**When** a client sends `GET https://malagaeventgear.com/my-post/`  
**Then** the CDN responds with HTTP `301`  
**And** the `Location` header value is `/blog/my-post/`

---

### SC-BR-02: Root-level page is NOT redirected

**Given** the same `_redirects` file is deployed  
**When** a client sends `GET https://malagaeventgear.com/packages/`  
**Then** the response status is NOT `301`  
**And** the SvelteKit application handles the request normally (HTTP `200` or `404` as appropriate)

---

### SC-BR-03: Redirect target returns 200

**Given** a valid migrated post with `slug: "my-post"` is prerendered at `/blog/my-post/`  
**When** a client follows the 301 redirect to `https://malagaeventgear.com/blog/my-post/`  
**Then** the response status is `200`

---

### SC-BR-04: `deriveOldPath` strips domain and preserves trailing slash

**Given** a WP `link` value of `"https://malagaeventgear.com/vintage-chairs/"`  
**When** `deriveOldPath("https://malagaeventgear.com/vintage-chairs/")` is called  
**Then** the return value is `"/vintage-chairs/"`

---

### SC-BR-05: `deriveOldPath` appends missing trailing slash

**Given** a WP `link` value of `"https://malagaeventgear.com/vintage-chairs"` (no trailing slash)  
**When** `deriveOldPath("https://malagaeventgear.com/vintage-chairs")` is called  
**Then** the return value is `"/vintage-chairs/"` (trailing slash appended)

---

### SC-BR-06: `deriveOldPath` throws on invalid input

**Given** the input string is `"not-a-url"` (not a valid absolute URL)  
**When** `deriveOldPath("not-a-url")` is called  
**Then** a `TypeError` (or equivalent error) is thrown  
**And** no empty string is returned silently

---

### SC-BR-07: Migration script generates correct rule format

**Given** a post with WP `link` `"https://malagaeventgear.com/birthday-party-decor/"` and `slug` `"birthday-party-decor"`  
**When** the migration script runs and generates `static/_redirects`  
**Then** the file contains the line `/birthday-party-decor/  /blog/birthday-party-decor/  301`  
**And** the line appears inside the `# BEGIN wp-blog-redirects` / `# END wp-blog-redirects` marker block

---

### SC-BR-08: Migration script is idempotent

**Given** the migration script has already generated `static/_redirects` with 75 post rules  
**When** the migration script is run again with the same post list  
**Then** `static/_redirects` contains the same 75 post rules (no duplicates, no omissions)

---

### SC-BR-09: Unrelated redirect rules are preserved

**Given** `static/_redirects` already contains the unrelated rule `/old-landing/  /packages/  301` outside the managed block  
**When** the migration script runs and generates the blog redirect block  
**Then** the unrelated rule `/old-landing/  /packages/  301` is still present in the file  
**And** the managed block contains only the blog-post rules

---

### SC-BR-10: Rule count warning when over 100

**Given** the sum of post redirects plus taxonomy redirects would exceed 100 rules  
**When** the migration script runs  
**Then** a warning is printed to stdout (e.g. `"WARNING: redirect count exceeds 100; taxonomy redirects omitted"`)  
**And** taxonomy redirect rules are NOT written to `static/_redirects`  
**And** all 75 post redirect rules ARE written

---

### SC-BR-11: Taxonomy archive redirect (SHOULD — if confirmed)

**Given** the WP REST API confirms a category with base `/category/` and `slug: "decoracion"`  
**And** the total rule count remains below 100  
**When** the migration script runs  
**Then** `static/_redirects` contains the rule `/category/decoracion/  /blog/category/decoracion/  301`

---

## Unit-Testability Note

The `deriveOldPath` function (SC-BR-04 through SC-BR-06) MUST be exported from the migration script as a named export and tested in isolation via Vitest, with no file-system or network dependencies. All other scenarios involving the full migration script output are integration-level and MAY be validated via script execution in CI.
