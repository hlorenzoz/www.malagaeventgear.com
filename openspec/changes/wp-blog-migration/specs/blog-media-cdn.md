# Spec: blog-media-cdn

## Capability

R2-backed image CDN for migrated WordPress media. Enriched `manifest.json` mapping every original WP image URL variant to its R2 counterpart. Images served exclusively from `cdn.malagaeventgear.com`.

---

## Delta: What MUST Be True After This Change

### 1. R2 Bucket and CDN Domain

- A Cloudflare R2 bucket named `meg-blog-media` MUST exist under account `cc26ab18f887fb1c63c19e17a0bb313f`.
- The custom domain `cdn.malagaeventgear.com` MUST be pointed at the `meg-blog-media` bucket.
- All objects in the bucket MUST be publicly accessible via `https://cdn.malagaeventgear.com/<key>`.
- The bucket MUST NOT be directly accessible via the `*.r2.cloudflarestorage.com` default URL in production (the CDN domain is the canonical URL).

### 2. Object Key Convention

- Each uploaded image MUST use the key format: `<wpId>/<fileName>` (e.g. `42/venue-main-1200x800.jpg`).
- Size variants (WordPress thumbnail, medium, large, full) MUST each be stored as separate objects with their respective filenames as the key suffix.
- The original full-size image MUST always be uploaded, even when intermediate sizes are also uploaded.

### 3. `manifest.json` Schema

A file `scripts/migrate-wp/manifest.json` MUST be generated by the migration script and committed to the repository. It serves as the authoritative mapping from original WP URLs to R2 URLs.

The manifest is a JSON object where each key is the **original WordPress URL** (absolute, as returned by the WP REST API) and each value is an object conforming to the following schema:

```
{
  "r2Url":              string,      // MUST — canonical CDN URL: https://cdn.malagaeventgear.com/<key>
  "category":           string,      // MUST — e.g. "blog", "venue", derived from WP attachment category or media library folder
  "tags":               string[],    // MUST — defaults to [] if WP media has no tags
  "wpId":               number,      // MUST — WordPress attachment post ID
  "fileName":           string,      // MUST — original file name as uploaded to WP (e.g. "venue-main.jpg")
  "title":              string,      // MUST — WP attachment title (may be empty string)
  "alt":                string,      // MUST — WP attachment alt text (may be empty string)
  "caption":            string,      // MUST — WP attachment caption (may be empty string)
  "description":        string,      // MUST — WP attachment description (may be empty string)
  "mimeType":           string,      // MUST — MIME type (e.g. "image/jpeg")
  "fileSize":           number,      // MUST — file size in bytes; 0 if unavailable from WP API
  "width":              number,      // MUST — pixel width of this variant
  "height":             number,      // MUST — pixel height of this variant
  "uploadedOn":         string,      // MUST — ISO 8601 date string of WP upload date
  "uploadedBy":         string,      // MUST — WP author login/display name
  "originalUrl":        string,      // MUST — the original WP URL (same as the manifest key, for self-documentation)
  "excludeFromSitemap": boolean      // MUST — true if the image should be excluded from any future image sitemap
}
```

Every WP image size variant (thumbnail, medium, medium_large, large, full, and any custom size registered by the WP theme) MUST have its own entry in the manifest, keyed by its own URL. Size variants of the same attachment share the same `wpId` and `fileName` (base) but differ in `r2Url`, `width`, `height`, and `fileSize`.

### 4. URL Rewriting in `.svx` Content

- After uploading all images, the migration script MUST rewrite every image URL inside each generated `.svx` file that points to the original WP domain to its corresponding `r2Url` from `manifest.json`.
- An `.svx` file that references an image URL absent from `manifest.json` MUST cause the migration script to exit with a non-zero status and log the unmapped URL.

### 5. `coverImage` Frontmatter Field

- The `coverImage` field in each migrated post's frontmatter MUST contain the `r2Url` of the full-size featured image.
- If a post has no featured image in WordPress, `coverImage` MUST be set to the site default OG image URL (as defined in `siteConfig`).

### 6. No Binary Assets in Git

- Image files MUST NOT be committed to the git repository.
- Only `manifest.json` (a text/JSON file) is committed.
- `.svx` files contain only CDN URLs, never base64 or blob data.

## Acceptance Scenarios

### SC-CDN-01: R2 URL resolves publicly

**Given** an image has been uploaded to `meg-blog-media` at key `42/venue-main.jpg`  
**When** an HTTP GET request is made to `https://cdn.malagaeventgear.com/42/venue-main.jpg`  
**Then** the response status is 200  
**And** the `Content-Type` header is `image/jpeg`

---

### SC-CDN-02: Manifest covers all WP image variants

**Given** the migration script has completed for a WP attachment with ID `42` that has 5 size variants  
**When** `manifest.json` is inspected  
**Then** exactly 5 entries exist with `wpId: 42`  
**And** each entry has a distinct `r2Url`  
**And** each `r2Url` begins with `https://cdn.malagaeventgear.com/`

---

### SC-CDN-03: Manifest entry contains all required fields

**Given** any entry in `manifest.json`  
**When** the entry is parsed  
**Then** all fields (`r2Url`, `category`, `tags`, `wpId`, `fileName`, `title`, `alt`, `caption`, `description`, `mimeType`, `fileSize`, `width`, `height`, `uploadedOn`, `uploadedBy`, `originalUrl`, `excludeFromSitemap`) are present  
**And** `r2Url` is a valid HTTPS URL  
**And** `wpId` is a positive integer  
**And** `excludeFromSitemap` is a boolean

---

### SC-CDN-04: Migrated post body contains no original WP domain URLs

**Given** the migration script has produced all `.svx` files  
**When** all `.svx` files are scanned for occurrences of the original WordPress image domain  
**Then** zero matches are found

---

### SC-CDN-05: Unmapped image URL fails migration loudly

**Given** an `.svx` file being generated references an image URL not present in `manifest.json`  
**When** the URL-rewriting step runs  
**Then** the migration script exits with a non-zero status code  
**And** the unmapped URL is printed to stderr

---

### SC-CDN-06: `coverImage` in frontmatter resolves to CDN URL

**Given** a migrated `.svx` file with a featured image  
**When** the frontmatter is parsed  
**Then** the `coverImage` field value begins with `https://cdn.malagaeventgear.com/`

---

### SC-CDN-07: Post without featured image gets default cover image

**Given** a WordPress post has no featured image set  
**When** the migration script generates the `.svx` file  
**Then** the `coverImage` frontmatter field is set to the site default OG image URL (not empty and not a WP URL)

---

### SC-CDN-08: No image files exist in the git repository

**Given** the migration has completed  
**When** the git repository is inspected for binary image files (`.jpg`, `.jpeg`, `.png`, `.webp`, `.gif`) under `src/`  
**Then** zero image files are found

---

### SC-CDN-09: Manifest `originalUrl` matches its manifest key

**Given** any entry in `manifest.json` at key `K`  
**When** the entry's `originalUrl` field is read  
**Then** `originalUrl === K`
